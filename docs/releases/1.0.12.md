# ZeroModel 1.0.12

ZeroModel 1.0.12 closes the current monolithic 1.x package line before the project begins its package-oriented 2.x architecture.

This release preserves the existing public `zeromodel` API and the NumPy-only base dependency while incorporating the completed video action-set runtime, durable observation and provenance records, governed final access, and bounded release validation.

## Highlights

### Complete video action-set runtime spine

The video action-set work now follows the established runtime structure:

```text
ZeroModelRuntime
    ↓
VideoActionSetFacade
    ↓
VideoActionSetEngine
    ↓
services
    ↓
DTO-only Store protocol
    ↓
in-memory or SQLite persistence
```

The runtime includes identity, episode planning, observation materialization, provider measurement, verification, mutation audit, final access, publication, and reconstruction services.

### Database-first immutable records

The release promotes benchmark identities, episode plans, observations, matrix blobs, provider descriptors, and observation operation chains into immutable DTO aggregates.

SQLite-backed stores provide durable ownership and reconstruction while preserving the rule that ORM objects never cross Store boundaries.

### Observation provenance and replay

Materialized observations retain:

- the observation record;
- content-addressed matrix pixels;
- provider observation descriptors;
- ordered transformation operations;
- complete operation-chain identity;
- replay and verification bindings.

### Governed final access

The finalization spine now includes:

- dedicated schema authority;
- historical-authority verification;
- deterministic evidence evaluation;
- compare-and-swap reservation;
- service-owned decision and digest computation;
- atomic filesystem publication;
- publication receipts;
- failure-state reconstruction;
- explicit terminal-state behavior.

### Bounded release validation

The repository contains a synthetic-only finalization integration package that validates:

- schema and authority separation;
- SQLite transaction and concurrency behavior;
- authorized observation ownership;
- historical authority;
- evaluator determinism;
- publication and failure injection;
- fresh-process reconstruction;
- Python and PowerShell operator commands;
- installed-wheel package boundaries.

The validation deliberately excludes preserved scientific fixtures and real provider measurements. Passing it establishes infrastructure behavior only; it does not promote scientific claims.

## Compatibility

- Distribution: `zeromodel`
- Version: `1.0.12`
- Python: `3.10`, `3.11`, and `3.12`
- Required runtime dependency: NumPy
- Optional persistence dependency: SQLAlchemy 2.x
- Existing 1.0 artifact identities and public import paths remain unchanged.

## Install

```bash
python -m pip install zeromodel==1.0.12
```

## Release boundary

ZeroModel 1.0.12 does not claim that the current visual providers solve open-world perception, that final-access infrastructure validates scientific provider quality, or that the existing hierarchy establishes planet-scale navigation.

The public claims audit remains the source of truth for the difference between implemented infrastructure, measured behavior, bounded validation, and future research.

## What comes next

The next development line will begin as `2.0.0.dev0` and reorganize ZeroModel into separately installable runtime packages. The first objective is architectural separation rather than new scientific claims:

```text
zeromodel core
    +
observation runtime
    +
video runtime
    +
provenance runtime
    +
persistence adapters
```

The 1.0.12 release provides the stable checkpoint from which that package decomposition can begin.
