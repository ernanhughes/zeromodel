## Disclosed provenance defect

The PowerShell runner used for the local run recorded `$device` in `command.txt`
and `runtime.txt` without defining that variable, while the executed command
hardcoded `--device cuda`.

Therefore the original command/device text generated by that script is not a
faithful provenance record.

The aggregate benchmark summary remains the observed console result. The
command-recording defect must not be concealed or reconstructed as if it had
been captured correctly at run time.

The corrected `scripts/benchmark.ps1` in this PR:

- defines the device once;
- executes from the same argument array it records;
- captures the evaluated Git commit and dirty state;
- records Python, NumPy, PyTorch, Transformers, CUDA, and GPU details;
- writes the exact argv separately;
- hashes the generated report;
- fails when the expected report is absent.

## Threshold interpretation

The v1 run used calibration quantile `0.0`, the minimum correct-row calibration
score. This is the most permissive threshold supported by the calibration
examples: it minimizes benign refusal but does not minimize false acceptance.

Consequently:

- the reported FAR and FRR values are one operating point, not complete system
  characteristics;
- systems C, D, and G show poor exact-row fidelity at this permissive point;
- system B remains open as a possible high-precision, lower-coverage Level 1
  reader until its full operating curve is measured;
- selecting a threshold on the existing evaluation cases would reuse the test
  set, so any promoted operating point requires an independent rejection-
  calibration split and untouched evaluation families.
